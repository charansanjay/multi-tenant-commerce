# Database Migrations Setup Guide

**Project:** Multi-Tenant-Commerce  
**Environment:** Windows + pnpm monorepo + Supabase CLI (Scoop)  
**Status:** Completed ✅

## Overview

This guide covers setting up and running the database schema migrations for the
first time on a fresh local Supabase instance. It documents the exact steps,
errors encountered, and how they were resolved.

## Prerequisites

- Docker Desktop installed and running
- Scoop installed (Windows package manager)
- pnpm installed
- Monorepo scaffolded with `supabase/` folder initialised at the root

## Step 1 — Place Migration Files

Create the migrations folder inside the `supabase/` directory at the monorepo
root (it does not exist by default after `supabase init`):

```text
platform/
└── supabase/
    ├── config.toml
    ├── seed.sql
    └── migrations/          ← create this folder
        ├── 0001_shared_utilities.sql
        ├── 0002_tenants.sql
        ├── 0003_staff_profiles.sql
        ├── 0004_customers.sql
        ├── 0005_addresses.sql
        ├── 0006_categories.sql
        ├── 0007_products.sql
        ├── 0008_product_variants.sql
        ├── 0009_coupons.sql
        ├── 0010_orders.sql
        ├── 0011_order_items.sql
        ├── 0012_payments.sql
        ├── 0013_audit_logs.sql
        ├── 0014_notifications.sql
        └── 0015_jwt_auth_hook.sql
```

Copy all 15 `.sql` migration files into `supabase/migrations/`.

> **Note on file naming:** The Supabase CLI applies migrations in alphabetical
> order. These files use a `0001_` prefix which always sorts before any future
> timestamp-named migrations generated by `supabase migration new`.

## Step 2 — Update the Supabase CLI

The Supabase CLI must be up to date or it will fail to parse newer keys in
`config.toml` with errors like:

```text
failed to parse config: decoding failed due to the following error(s):
'db' has invalid keys: health_timeout
```

**On Windows with Scoop (correct approach):**

```bash
scoop update supabase
```

Verify the version:

```bash
supabase --version
# should be 2.84.2 or higher
```

> **Do not** use `pnpm add -D supabase -w` on Windows — the npm package
> fails to install the `.EXE` binary on Windows and does not work.
> The Scoop global install is the correct approach on Windows.

**Update the scripts in root `package.json`** to use the global CLI directly
(drop `pnpm dlx`):

```json
{
  "scripts": {
    "db:start":  "supabase start",
    "db:stop":   "supabase stop",
    "db:reset":  "supabase db reset",
    "db:status": "supabase status",
    "db:migrate":"supabase migration new",
    "db:push":   "supabase db push",
    "db:types":  "supabase gen types typescript --local > apps/admin/src/types/database.types.ts"
  }
}
```

## Step 3 — Configure the JWT Auth Hook in config.toml

The `custom_jwt_claims` PostgreSQL function (defined in `0015_jwt_auth_hook.sql`)
must be registered as the Supabase Auth hook. For local development this is done
in `config.toml` — **not** in the Studio UI (the Hooks section only exists in the
cloud dashboard).

Find this commented-out section in `supabase/config.toml`:

```toml
# [auth.hook.custom_access_token]
# enabled = true
# uri = "pg-functions://<database>/<schema>/<hook_name>"
```

Uncomment and fill it in exactly as follows:

```toml
[auth.hook.custom_access_token]
enabled = true
uri = "pg-functions://postgres/public/custom_jwt_claims"
```

> **Important:** The URI must say `public` — not the literal placeholder
> `<schema>`. Leaving `<schema>` causes the auth container to crash with
> `invalid schema name: <schema>`.

## Step 4 — Start Supabase and Apply Migrations

```bash
pnpm db:start
```

Once running, apply all migrations:

```bash
pnpm db:reset
```

This wipes the local database and re-runs all 15 migrations in order from
scratch. It is the correct command during early development before any real
data exists.

Verify in Supabase Studio at `http://localhost:54323` → Table Editor that all
13 tables are present.

## Step 5 — Generate TypeScript Types

```bash
pnpm db:types
```

This overwrites `apps/admin/src/types/database.types.ts` with types generated
directly from the live local schema. The generated file is the authoritative
source of types going forward — do not hand-edit it.

## Troubleshooting — Errors Encountered

### Error: `failed to parse config` — invalid keys

**Cause:** Supabase CLI version too old for the `config.toml`.  
**Fix:** Update CLI via `scoop update supabase`.

### Error: `relation "staff_profiles" does not exist`

**Cause:** The `custom_jwt_claims` function was in `0001_shared_utilities.sql`
but it references `staff_profiles%ROWTYPE` and `tenants%ROWTYPE`. PostgreSQL
resolves these types at function creation time, not at call time — so the
tables must already exist.  
**Fix:** Moved `custom_jwt_claims` to `0015_jwt_auth_hook.sql` which runs
after all tables are created.

### Error: `invalid schema name: <schema>`

**Cause:** The `config.toml` JWT hook URI was left with the literal placeholder
`<schema>` instead of `public`.  
**Fix:** Set `uri = "pg-functions://postgres/public/custom_jwt_claims"`.

### Error: `container name already in use`

**Cause:** `supabase stop` removes containers but leaves orphaned ones behind,
preventing `supabase start` from recreating them.  
**Fix:** Remove orphaned containers manually:

```bash
docker ps -a --filter "name=multi-tenant-commerce" --format "{{.Names}}" | xargs docker rm -f
```

Then run `pnpm db:start` again.

### Error: `supabase start is not running` when running `db:reset`

**Cause:** Tried to run `db:reset` before `db:start`.  
**Fix:** Always run `pnpm db:start` first, then `pnpm db:reset`.

## Day-to-Day Workflow

### Making a schema change

```bash
# 1. Create a new migration file (generates a timestamped file)
pnpm db:migrate add_column_to_orders

# 2. Write your SQL in the generated file at supabase/migrations/

# 3. Apply locally (wipes and reruns all migrations)
pnpm db:reset

# 4. Regenerate types
pnpm db:types

# 5. Push to remote when ready
pnpm db:push
```

### Starting a dev session

```bash
pnpm db:start    # start local Supabase
pnpm dev         # start Next.js
```

### Ending a dev session

```bash
pnpm db:stop     # stop local Supabase containers
```

## Cloud / Remote Setup (When Ready to Deploy)

This is done once when deploying to production — not during local development.

### 1. Link to remote project

```bash
supabase link --project-ref <your-project-ref>
```

### 2. Push migrations to remote

```bash
pnpm db:push
```

### 3. Register the JWT Auth Hook in cloud dashboard

Supabase Dashboard → your project → **Authentication → Hooks**  
→ Custom Access Token → select `custom_jwt_claims`

> This is the cloud equivalent of the `config.toml` setting used locally.
> You must do both — `config.toml` for local, Dashboard UI for remote.

### 4. Regenerate types from remote schema

```bash
supabase gen types typescript --project-id <your-project-ref> > apps/admin/src/types/database.types.ts
```
